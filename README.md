# Puzzles

A suite of daily math and word puzzles built with React and Vite, deployed to GitHub Pages. Each puzzle type generates one new challenge per day, keyed to Pacific Time.

## Games

| Game            | Description                                                                     |
| --------------- | ------------------------------------------------------------------------------- |
| **All Ten**     | Make each target from 1 to 10 using four starting numbers and basic arithmetic. |
| **Scurry**      | Place bugs to fill every target square on the grid.                             |
| **Clueless**    | Complete six crossing words without any clues. Three difficulties per day.      |
| **Folds**       | Reflect triangles to match the target pattern.                                  |
| **Honeycombs**  | Fill each honeycomb to form a connected path from 1 to 10.                      |
| **Sum Tiles**   | Slide tiles so every row and column hits its target sum.                        |
| **Productiles** | Slide tiles so every row and column hits its target product.                    |

## Local Development

**Prerequisites:** Node.js 20+

```bash
npm ci
npm run dev
```

The hub runs at [http://localhost:5173/](http://localhost:5173/). Individual games are served at their own paths, e.g. `/puzzlegames/allten/`.

> `npm run dev` runs a pre-dev boundary check automatically. If you see a check failure on startup, see [docs/allten-migration/](docs/allten-migration/).

## Build

```bash
npm run build
```

Output goes to `dist/`. The build runs the boundary check automatically before bundling.

### GitHub Pages deployment

The base path is resolved in this order:

1. `VITE_BASE_PATH` environment variable (e.g. `VITE_BASE_PATH=/puzzles/`)
2. The repo slug from `GITHUB_REPOSITORY` (set automatically in GitHub Actions)
3. Falls back to `/puzzles/`

To deploy manually:

```bash
npm run deploy
```

This builds and publishes the `dist/` folder to the `gh-pages` branch via [gh-pages](https://github.com/tschaub/gh-pages).

## Quality Checks

```bash
npm run check:migration   # full gate: boundary + header parity + puzzle schemas + Vitest + All Ten Jest + build
npm run check:allten-boundary   # All Ten / suite integration boundary
npm run check:header-parity     # shared chrome contract across all games
npm run check:puzzle-schemas    # required keys on each game's puzzles.js tiers
npm run compact:puzzles         # re-compact folds/sumtiles/productiles/honeycombs data (one line per puzzle)
npm run lint                    # ESLint: suite .js/.jsx + root tooling .mjs (Node)
npm run format                  # Prettier — suite, shared-contracts, tools (excludes src/allten/runtime/)
npm run format:check            # CI-friendly Prettier check
```

`src/allten/runtime/` keeps its own Prettier and ESLint; run `npm --prefix src/allten/runtime run lint` / `pretty-check` there.

Four large `puzzlegames/…/puzzles.js` files are listed in `.prettierignore` so Prettier does not reformat them. Folds uses three lines per puzzle (`start`, `target`, `folds`) with a blank line between puzzles; the other three use one line per puzzle. Run `npm run compact:puzzles` after bulk edits if layout drifts.

CI runs `check:migration` on every push and pull request (see [`.github/workflows/migration_safety.yml`](.github/workflows/migration_safety.yml)).

## Clueless Puzzle Pipeline

Clueless puzzles are generated by a Python pipeline under `tools/clueless/`. Scripts require Python 3.10+. Optional third-party packages (word frequency tooling, MILP for combined reports) are listed in `tools/clueless/requirements-wordfreq.txt`—install them when you run those helpers; many pipeline steps use only the standard library.

```bash
pip install -r tools/clueless/requirements-wordfreq.txt   # optional, for zipf/MILP tooling
npm run generate:clueless-puzzles
```

See the individual script headers in `tools/clueless/` for pipeline ordering and options.

## Contributing

This project uses a structured migration approach for the All Ten game, which runs its own TypeScript/Webpack/Jest subpackage at `src/allten/runtime/` while the rest of the suite uses Vite. Shared suite modules live under `shared-contracts/` and are imported as `@shared-contracts/…` (Vite, Vitest, and the All Ten Webpack/Jest configs resolve this alias).

Before making changes that touch the All Ten integration boundary, read:

- [`docs/allten-migration/safe-shared-boundary.md`](docs/allten-migration/safe-shared-boundary.md) — what crosses the boundary and what must not
- [`docs/allten-migration/regression-checklist.md`](docs/allten-migration/regression-checklist.md) — manual verification steps
- [`docs/allten-migration/storage-compatibility-contract.md`](docs/allten-migration/storage-compatibility-contract.md) — localStorage key conventions
- [`docs/allten-migration/stats-and-toolchain-criteria.md`](docs/allten-migration/stats-and-toolchain-criteria.md) — criteria for retiring the legacy Webpack build
