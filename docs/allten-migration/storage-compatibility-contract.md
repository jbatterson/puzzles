# All Ten Storage Compatibility Contract

This document defines the storage behavior that must not change during migration.

## Scope

- Runtime: root entry `src/allten/main.jsx` with app state from `src/allten/runtime/src/state/AppState.ts`.
- Source of truth: browser `localStorage` on the current origin.
- Legacy compatibility: selected cookie fields are read and merged on startup.

## Required Keys and Shapes

### 1) Profile key

- Key: `allten-profile`
- Writer: `saveToStorage()` in `src/allten/runtime/src/state/AppState.ts`
- Reader: `loadFromStorage()` in `src/allten/runtime/src/state/AppState.ts`
- Value: JSON object compatible with:
  - `numPlays: number`
  - `numStreak: number`
  - `mostRecentPlayDate?: string` (format `yyyymmdd`)
  - `numAllTens: number`
  - `mostRecentAllTen?: string` (format `m/d/yyyy`)
  - `showHelpOnStart: boolean`

### 2) Daily problem key

- Key pattern: `${pstStringify(problemDate)}-problem`
- Example: `3/26/2026-problem`
- Writer: `saveToStorage()`
- Reader: `loadFromStorage()` (today and yesterday)
- Value: JSON-serialized `Problem` object generated by `createProblemForDate(...)`.

### 3) Daily targets key

- Key pattern: `${pstStringify(problemDate)}-targets`
- Example: `3/26/2026-targets`
- Writer: `saveToStorage()`
- Reader: `loadFromStorage()` (today and yesterday)
- Value: JSON array of target states (`TargetState[]`) including solution metadata.

## Startup Lifecycle Contract

In order, startup behavior must remain:

1. `AppState.loadFromStorage()` is called once after `new AppState(...)`.
2. `loadFromCookies()` may hydrate profile fields (legacy compatibility).
3. Yesterday and today values are read from `localStorage`.
4. Target data is validated against current problem.
5. Profile is read from `allten-profile`.
6. `updateProfileStatsToToday()` may reset streak to `0` when appropriate.
7. `validateDaysPlayed()` and `validateNumAllTens()` reconcile historic count bugs.
8. `cleanupOldStorageEntries()` runs and keeps:
   - `allten-profile`
   - last 30 days of `-problem` and `-targets` keys

## Mutation Contract During Play

When first valid solution is submitted for a day:

- `numStreak += 1`
- `numPlays += 1`
- `mostRecentPlayDate = pstStringifyForRobots(problemDate)`
- `saveToStorage()` is called

When first completion of all 10 targets for a day:

- Results modal is shown
- `numAllTens += 1`
- `mostRecentAllTen = pstStringify(problemDate)`
- `saveToStorage()` is called again

## Date/Key Format Contract

- `pstStringify(date)` produces `m/d/yyyy` (used in storage key names).
- `pstStringifyForRobots(date)` produces `yyyymmdd` (used by streak logic).
- Cleanup logic only deletes date-shaped keys matching:
  - suffix `-problem` or `-targets`
  - date regex `^\\d{1,2}/\\d{1,2}/\\d{4}$`

## Migration Guardrails

Do not change, rename, or remap these without a dedicated migration:

- `allten-profile` key name
- `m/d/yyyy-problem` and `m/d/yyyy-targets` key pattern
- `yyyymmdd` comparison semantics for `mostRecentPlayDate`
- startup call order around validation and cleanup

If migration is needed, implement dual-read + single-write with rollback support first.
